## Goal

New POST endpoint that lets an active affiliate create a campaign
anchored on a general `service_templates` row (no assignment required).
Parallel to the existing
`POST /api/affiliate/assignments/[id]/campaigns` for per-diviner
campaigns.

## Endpoint spec

**File:** `src/app/api/affiliate/general-campaigns/route.ts` (new)

### `POST /api/affiliate/general-campaigns`

**Auth:** authed user resolved through
`resolveAffiliateForCaller(admin, user.id)`. Account must exist and be
`status='active'`. Otherwise 403.

**Request body:**
```ts
{
  service_template_id: string;       // required, must be general + enabled
  name: string;                      // required, ≤ 120 chars
  description?: string;              // optional, ≤ 1000
  channel?: string;                  // optional, ≤ 50
  utm_source?: string;
  utm_medium?: string;
  utm_campaign?: string;
}
```

**Validation:**
- `service_template_id` resolves to a row where
  `is_general = true AND affiliate_program_enabled = true`. Otherwise
  422 with detail `template_not_eligible`.
- `name.trim()` non-empty.
- Rate-limit: 20/hour per affiliate account (mirrors the per-diviner
  endpoint).

**Insert:**
```ts
{
  owner_type: 'affiliate',
  owner_affiliate_type: 'general',
  owner_affiliate_id: null,
  owner_affiliate_account_id: ctx.account.id,
  source_assignment_id: null,
  diviner_id: null,
  destination_type: 'SERVICE',
  destination_service_template_id: service_template_id,
  campaign_code: <auto-generated cmp_+8chars>,
  share_url: `${APP_URL}/r/<code>`,
  status: 'active',
  start_date: NOW(),
  // commission_*_snapshot — NOT set; the stamp comes from
  //   service_templates at booking time, not from the campaign.
  name, description, channel, utm_*, ...
}
```

After insert, also insert a matching `tracking_links` row keyed on the
generated code.

**Response:**
```ts
// 201
{
  data: {
    campaign_id: string,
    campaign_code: string,
    share_url: string,
  }
}
```

**Errors (RFC 9457 problem+json):**
- 401 — not authenticated
- 403 — `Not an active affiliate`
- 422 `template_not_eligible` — template doesn't exist, or isn't general,
  or `affiliate_program_enabled=false`
- 422 `validation_error` — name missing/too long, etc.
- 429 — rate limit exceeded
- 500 — DB error

## Idempotency

Same affiliate posting the same `service_template_id` twice creates TWO
campaigns. That's intentional (UTM variants, A/B tests). The Marketing
Kit may de-dupe at display time but the endpoint should not.

## Archive flow — existing endpoint NEEDS a tweak (verified 2026-04-28)

The existing `PATCH /api/affiliate/campaigns/[id]/route.ts` reuses the
same code path for general campaigns, BUT pre-flight reading revealed
a hardcoded ownership check at lines 78-86:

```ts
// CURRENT (only matches per-diviner):
if (
  !campaign ||
  campaign.owner_type !== "affiliate" ||
  campaign.owner_affiliate_type !== "diviner_affiliate" ||  // ← rejects 'general'
  !ctx.junctionIds.includes(campaign.owner_affiliate_id as string)
) {
  return problem(404, "Campaign not found");
}
```

This rejects general campaigns even when the caller owns them. **Update
this check** as part of Phase 1.5:

```ts
// REPLACEMENT (matches both per-diviner and general):
const isOwnedPerDiviner =
  campaign.owner_affiliate_type === "diviner_affiliate" &&
  ctx.junctionIds.includes(campaign.owner_affiliate_id as string);

const isOwnedGeneral =
  campaign.owner_affiliate_type === "general" &&
  campaign.owner_affiliate_account_id === ctx.account.id;

if (
  !campaign ||
  campaign.owner_type !== "affiliate" ||
  (!isOwnedPerDiviner && !isOwnedGeneral)
) {
  return problem(404, "Campaign not found");
}
```

Also extend the SELECT at line 73-76 to pull `owner_affiliate_account_id`:

```ts
.select("id, status, owner_type, owner_affiliate_id, owner_affiliate_type, owner_affiliate_account_id")
```

This change ships AS PART OF this task (04), since it's a precondition
for the affiliate to archive a general campaign. Add it to the
acceptance criteria below.
