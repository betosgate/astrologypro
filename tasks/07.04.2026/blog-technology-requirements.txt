Blog Technology / Content Publishing Module

Requirement Analysis Document

Perspective: Senior Solution Architect

1) Objective

The blog module must not behave like a simple "post listing page." It should function as a full publishing platform that supports:

* content creation and editorial workflow
* high SEO performance
* authority building for diviners/experts
* product, course, and service conversion support
* affiliate/share tracking
* long-term scalability for content growth
* structured analytics for content performance

This module should support both marketing goals and knowledge-delivery goals.

For this platform, the blog should help with:
* astrology education content
* forecast and transit updates
* diviner thought leadership
* course promotion
* product promotion
* internal linking to offerings
* audience capture through newsletter/lead magnet/CTA flows

---

2) Core Goals

The blog system must be designed to achieve the following:

1. Drive organic traffic
   * rank for educational and intent-based content
   * support SEO-friendly structure and schema

2. Build trust and authority
   * showcase diviner expertise
   * allow author profile pages and authored content

3. Support conversions
   * promote courses, services, sessions, and products from within blog content
   * support targeted CTA blocks

4. Support editorial operations
   * allow admins and diviners to draft, review, publish, and update content safely

5. Support future scale
   * categories, tags, series, taxonomy, multilingual readiness, analytics, and structured search must be built cleanly from day one

---

3) User Roles

3.1 Admin

Can:
* create, edit, publish, unpublish, archive any post
* manage categories, tags, series, authors, SEO metadata
* assign diviner as author
* manage featured posts and homepage placements
* manage blog CTA blocks
* manage redirects and slug history
* view analytics dashboard
* approve/reject content submitted by diviners
* configure blog settings

3.2 Diviner / Author

Can:
* create own draft posts
* edit own drafts
* submit draft for review
* preview unpublished content
* manage own author bio/profile content
* attach related services/courses/products if permission is granted

Cannot by default:
* publish directly unless special permission is enabled
* delete published content permanently
* override SEO global rules

3.3 Editor / Content Manager

Can:
* edit and refine posts from multiple authors
* manage editorial quality
* optimize metadata, readability, structure, and internal links
* schedule publishing

3.4 Affiliate

Can:
* access public blog content
* share blog articles using their own tracked link
* optionally have article-level referral tracking for linked products/courses

Cannot:
* author/edit posts unless explicitly granted a separate role

3.5 Public Visitor

Can:
* browse blog homepage
* view post details
* search/filter content
* share articles
* subscribe to newsletter
* click related product/course/service CTAs
* explore author/diviner profile pages

16) Recommended Technical Architecture

16.1 Frontend Rendering
* Next.js App Router
* hybrid rendering strategy:
  * static generation for published blog pages where possible
  * ISR / on-demand revalidation after publish/update
  * SSR only where truly needed

16.2 CMS Approach

Preferred for this product: custom content management module inside the same Next.js full-stack system

Why:
* tighter integration with roles like admin/diviner/affiliate
* easier product/course linking
* cleaner permission control
* better long-term business logic ownership
* avoids overdependence on external CMS for core workflows

Important architect note:
Use MDX only for developer-managed static content if needed.
For business-authored blog content, prefer structured JSON block content stored in DB and rendered through controlled components. That gives:
* safer rendering
* better versioning
* better validation
* better no-code editing
* cleaner future extensibility

16.3 Suggested backend components

* Next.js server actions / route handlers or dedicated internal API layer
* PostgreSQL for structured content metadata
* object storage for media
* CDN for delivery
* queue/background job support for:
  * scheduled publishing
  * sitemap rebuilds
  * search indexing
  * image processing
  * analytics aggregation

19) Security Requirements

* RBAC must be enforced on all admin/editor actions
* draft and preview routes must be protected
* unpublished content must never be publicly indexable
* slug collisions must be prevented
* HTML/content sanitization required
* file uploads must be validated
* image/file upload malware and type checks should exist where applicable
* audit logs required for publish/unpublish/delete actions
* rate limiting required on search and newsletter endpoints
* secure handling of affiliate/referral parameters required

---

20) Performance Requirements

* blog detail pages must load fast and be SEO-first
* featured image and content assets must be optimized
* use lazy loading for non-critical blocks
* use CDN caching for public blog assets
* use ISR or on-demand revalidation after content publish/update
* search queries must be indexed properly
* avoid client-heavy rendering for core content
* maintain excellent Core Web Vitals on blog pages

24) Gold-Standard Acceptance Criteria

This module should be considered complete only when:
* admins and diviners can manage content safely through controlled workflows
* published articles render fast and SEO-friendly
* author pages and taxonomy pages work cleanly
* content supports conversion blocks for courses/products/services
* blog URLs remain stable and redirects handle slug changes
* analytics can show what content performs best
* the module is scalable enough to support hundreds or thousands of posts without structural rewrite

---

25) Strong Architect Recommendation

For this platform, I would strongly recommend:
* Next.js App Router
* custom blog CMS inside the product
* structured block editor
* PostgreSQL-backed content model
* object storage for media
* ISR + on-demand revalidation
* rich SEO + redirect system from day one
* tight integration with diviner, course, product, and affiliate modules

That gives the cleanest long-term result.
