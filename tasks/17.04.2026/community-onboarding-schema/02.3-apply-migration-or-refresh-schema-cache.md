## Goal

Fix the database-side cause of the `birth_country` schema cache error.

## Path A: Column Is Missing

Apply the existing migration or equivalent SQL:

```sql
ALTER TABLE community_members
  ADD COLUMN IF NOT EXISTS birth_country VARCHAR(100);
```

Then verify:

```sql
SELECT column_name, data_type, character_maximum_length, is_nullable
FROM information_schema.columns
WHERE table_schema = 'public'
  AND table_name = 'community_members'
  AND column_name = 'birth_country';
```

## Path B: Column Exists but API Still Fails

Refresh Supabase/PostgREST schema cache:

```sql
NOTIFY pgrst, 'reload schema';
```

If the deployed app is a long-running process, restart the Next.js server after the cache reload.

## Important Rules

- Do not create a duplicate column with a different name.
- Do not rename `birth_country` to `country`, `birthCountry`, or another variant.
- Keep the field nullable.
- Do not remove the field from the API payload to hide the error.

## Acceptance Criteria

- [ ] If the column was missing, it has been added to `community_members`.
- [ ] If the column already existed, the schema cache was refreshed.
- [ ] The Next.js server was restarted if required by the target environment.
- [ ] A follow-up schema check confirms the column is visible after remediation.
