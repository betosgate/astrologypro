# Task 07.1: Product Category API - CRUD Endpoints and Schema - 2026-04-07

- Status: Planned
- Priority: P1
- Scope: Database schema, CRUD API, Category detail
- Task File: `tasks/perennial/2026-04-07/07.1-product-category-api-crud-endpoints-and-schema.md`

## Objective

Create the back-end infrastructure to support the Product Category management functionality. This includes defining the database table and building the unified Add/Edit API, Delete API, and Fetch APIs (List and Detail).

## Schema Fields (Matches Shared Reference Image 3)

- `id`: UUID (Primary Key)
- `title`: Text (Required)
- `description`: Text (Required)
- `priority`: Integer (Required)
- `status`: Text (Active/Inactive)
- `image_url`: Text (S3/Cloudinary/Storage path)
- `created_at`: Timestamptz (Auto)
- `updated_at`: Timestamptz (Auto)

## Implementation Steps

1.  **Database Migration**:
    - Create a SQL migration to add the `perennial_product_categories` table.
    - Set default values for priority and status if needed.
    - Add an `updated_at` auto-update trigger.

2.  **Unified Post/Put API (`/api/perennial/product-categories`)**:
    - If `id` exists in the body, perform `UPDATE`.
    - Otherwise, perform `INSERT`.
    - Ensure title and priority are present.

3.  **Delete API (`/api/perennial/product-categories/[id]`)**:
    - Method: `DELETE`.
    - Secure the endpoint to ensure only admins can delete.

4.  **Fetch APIs**:
    - **List (`GET /api/perennial/product-categories`)**: Returns all categories for the list view, sorted by priority.
    - **Detail (`GET /api/perennial/product-categories/[id]`)**: Returns full details for a single category for the edit form.

---

## Done Definition

- Database table is correctly provisioned.
- `POST`/`PUT` endpoints return 200/201 on success.
- `DELETE` endpoint removes the category (handle potential product associations).
- `GET` endpoints return valid category JSON.
- API authentication is active.
