## Goal

Define and implement how post-login profile-completion data is saved to database for the authenticated Perennial user.

## Core Rule

1. Save must update by authenticated `user_id` from session/auth context.
2. Do not trust a client-sent `user_id` in payload.

## API Surface

1. Use existing authenticated Perennial profile API surface.
2. No separate anonymous/public save API.
3. Support incremental save (partial update / patch behavior).

## Payload (request body)

Payload keys are form fields:

1. `firstName`
2. `lastName`
3. `email`
4. `phone`
5. `gender`
6. `occupation`
7. `address`
8. `city`
9. `state`
10. `zip`
11. `date_of_birth`
12. `birth_time`
13. `birth_location_label`
14. `birth_lat`
15. `birth_lng`
16. `birth_tzone`
17. `relationship_status`
18. `mainConcern`
19. `longTermGoals`
20. `personality`
21. `strengths`
22. `lifeAreasFulfilling`
23. `lifeAreasImprovement`
24. `majorLifeEvents`
25. `relationship_with_family`
26. `focus_on_specific_relationships`
27. `guidance_on_specific_decision`
28. `concerns_about_romantic_life`

## Read/Write Flow

1. On form open, API GET returns profile data for authenticated `user_id`.
2. UI pre-fills values from GET.
3. On save, API writes updates for same authenticated `user_id`.
4. API returns updated profile object.

## Validation

1. Required completion fields must be validated on API.
2. Return structured errors for missing/invalid fields.
3. Persist completion flag/state for dashboard gating.

## Acceptance

1. Save always maps to logged-in user only.
2. Cross-user overwrite is not possible via payload tampering.
3. Partial saves work and are persisted.
4. GET and save responses keep UI in sync.
